## 왜 여러 로봇이 필요한가

단일 로봇의 물리적 한계(팔 길이, 이동 속도, 탑재 가능한 센서)는 많은 응용에서 병목이 됩니다. 이를 극복하기 위해 **다중 로봇 시스템(multi-robot system, MRS)** 이 사용됩니다. 대표적인 동기는 다음과 같습니다.

- **병렬 작업(parallelism)**: 여러 로봇이 동시에 다른 구역을 탐색하거나 서로 다른 조립 공정을 수행합니다.
- **물리적 협력(physical cooperation)**: 단일 로봇이 들 수 없는 무거운 물체를 여러 로봇이 함께 운반합니다.
- **내결함성(fault tolerance)**: 일부 로봇이 고장나도 나머지가 임무를 계속합니다.
- **공간적 커버리지(coverage)**: 수색 구조, 환경 모니터링처럼 넓은 공간을 효율적으로 커버해야 하는 경우.

## 제어 아키텍처: 중앙집중식 vs. 분산식

다중 로봇 시스템에서 가장 중요한 설계 결정은 **제어 아키텍처**입니다.

```text
┌──────────────────────────────────────────────────────────────────────┐
│            중앙집중식 vs. 분산식 제어 비교                            │
├──────────────────────────┬───────────────────────────────────────────┤
│ 중앙집중식               │ 분산식                                    │
│ (Centralized)            │ (Decentralized)                           │
├──────────────────────────┼───────────────────────────────────────────┤
│ 글로벌 최적 계획 가능    │ 통신 지연/단절에 강인                     │
│ 일관성 보장 쉬움         │ 개별 로봇 고장 시에도 동작 가능           │
│ 통신 병목(중앙 서버)     │ 전역 최적해 보장 어려움                   │
│ 중앙 서버 고장 시 전체   │ 협조 행동 설계 복잡                       │
│ 시스템 마비              │                                           │
│ 소규모 팀에 적합         │ 대규모 팀, 불안정 통신에 적합             │
└──────────────────────────┴───────────────────────────────────────────┘
```

실제 시스템에서는 두 방식을 결합한 **계층적(hierarchical) 아키텍처**가 많이 사용됩니다. 상위 계층에서 전체 임무를 계획하고, 각 로봇이 하위 계획을 자율적으로 실행합니다.

## 핵심 문제들

### 과제 할당(Task Allocation)

어떤 로봇이 어떤 과제를 수행할지 결정하는 문제입니다. 일반적으로 최적 할당은 NP-hard이므로 실시간 시스템에서는 휴리스틱(heuristic)이나 경매 기반(auction-based) 방법이 사용됩니다.

**시장 기반 방법(market-based approach)**: 과제를 경매에 부쳐 각 로봇이 자신이 가장 효율적으로 수행할 수 있는 과제를 낙찰받습니다. 통신이 가능한 환경에서 분산 방식으로 작동합니다.

### 충돌 회피와 경로 계획

여러 로봇이 같은 공간에서 이동할 때 경로가 충돌할 수 있습니다. **다중 에이전트 경로 계획(Multi-Agent Path Finding, MAPF)** 은 모든 로봇이 충돌 없이 목적지에 도달하는 경로 집합을 찾는 문제입니다.

CBS(Conflict-Based Search), 우선순위 기반(priority-based) 방법, 분산 반응형(reactive) 방법 등이 연구되어 있습니다. 로봇 수가 늘어날수록 계산 복잡도가 급증하므로, 대규모 팀에서는 근사 알고리즘이 불가피합니다.

### 통신 제약

물리 세계에서 통신은 신뢰할 수 없습니다. 지연(latency), 패킷 손실, 대역폭 제한, 통신 범위 밖으로의 이탈이 발생합니다. 실용적인 다중 로봇 시스템은 **통신 없이도 기본 안전 행동을 유지**할 수 있어야 합니다.

## 다중 에이전트 강화학습(MARL)

**MARL(Multi-Agent Reinforcement Learning)** 은 여러 에이전트가 공유 환경에서 동시에 학습하는 RL의 확장입니다.

### MARL의 핵심 도전

**신용 할당 문제(credit assignment)**: 팀이 공동으로 보상을 받을 때, 어떤 에이전트의 어떤 행동이 결과에 기여했는지 파악하기 어렵습니다. COMA(Counterfactual Multi-Agent) 등의 방법이 이를 해결하려 합니다.

**비정상성(non-stationarity)**: 다른 에이전트도 동시에 학습하므로, 각 에이전트 입장에서 환경이 계속 변화합니다. 단일 에이전트 RL이 가정하는 정상적(stationary) 환경 가정이 깨집니다. 이는 표준 RL 수렴 보장을 무효화합니다.

**확장성(scalability)**: 에이전트 수가 늘어나면 공동 행동 공간이 지수적으로 증가합니다. 파라미터 공유(parameter sharing), 역할 기반(role-based) 분해, 관심 기반 통신(attention-based communication) 등이 확장성을 높이는 데 활용됩니다.

### 협조(Cooperative) vs. 경쟁(Competitive) vs. 혼합(Mixed)

MARL은 에이전트 간의 관계에 따라 세 유형으로 나뉩니다.

```text
┌─────────────────────────────────────────────────────────────┐
│                  MARL 시나리오 유형                          │
├───────────────┬──────────────────┬──────────────────────────┤
│ 협조          │ 경쟁             │ 혼합                     │
├───────────────┼──────────────────┼──────────────────────────┤
│ 공동 보상     │ 제로섬(zero-sum) │ 팀 내 협조 + 팀 간 경쟁  │
│ 창고 피킹 팀  │ 게임 AI(바둑)    │ 전략 게임, 협상          │
│ 수색 구조     │ 경쟁 자원 획득   │ 공동 로봇 조작           │
└───────────────┴──────────────────┴──────────────────────────┘
```

로보틱스 응용에서는 대부분 협조 시나리오이며, **CTDE(Centralized Training, Decentralized Execution)** 패러다임이 주류입니다. 학습 중에는 모든 에이전트의 정보를 중앙에서 활용하여 협조 전략을 학습하지만, 실행 시에는 각 에이전트가 자신의 관측만으로 행동합니다.

## 스웜 로보틱스(Swarm Robotics)

**스웜 로보틱스**는 수십~수천 개의 단순한 로봇이 **국소적(local) 상호작용**만으로 집단 수준에서 복잡한 행동을 만들어내는 분야입니다. 곤충 군집(개미, 벌떼)의 행동에서 영감을 받았습니다.

### 창발(Emergence)의 원리

스웜의 핵심은 **창발(emergence)** 입니다. 개별 로봇은 간단한 규칙만 따릅니다.
- 가까운 이웃과 일정 거리 유지(분리, separation)
- 이웃의 평균 방향으로 이동(정렬, alignment)
- 이웃의 평균 위치 쪽으로 이동(응집, cohesion)

이 세 규칙만으로 새 떼처럼 유기적으로 움직이는 군집 행동이 나타납니다. 중앙 제어 없이도 복잡한 대형 형성, 경로 탐색, 물체 운반이 가능합니다.

### 스웜의 장점과 한계

**장점**: 개별 로봇의 사양이 낮아도 되므로 비용이 저렴합니다. 일부 로봇 고장 시에도 임무를 계속할 수 있어 강인성(robustness)이 높습니다. 필요에 따라 로봇 수를 늘려 성능을 확장할 수 있습니다.

**한계**: 개별 로봇의 행동만으로는 복잡한 임무 논리를 표현하기 어렵습니다. 전체 군집의 행동을 예측·디버깅하기 힘듭니다. 통신 범위가 제한되면 군집이 분리될 수 있습니다.

## 시뮬레이션에서의 다중 로봇 학습

NVIDIA Isaac Sim 같은 GPU 가속 시뮬레이터는 수천 개의 로봇 인스턴스를 병렬로 실행할 수 있어 MARL 연구에 중요한 기반이 됩니다. 수천 개 환경을 동시에 실행하면서 협조 정책을 빠르게 학습할 수 있습니다. 다만 시뮬레이션에서 학습한 협조 전략을 실물 다중 로봇 시스템으로 전이할 때는 단일 로봇보다 sim-to-real 갭이 더 복잡하게 나타납니다. 로봇 간 타이밍, 통신 지연, 물리적 상호작용 모두 시뮬레이션과 실물 간 차이를 만듭니다.

다중 로봇 시스템은 물리 AI의 응용 범위를 크게 넓히지만, 설계 복잡도와 검증 어려움도 그만큼 커집니다. 실용적인 시스템을 만들기 위해서는 로봇의 수와 사용 환경에 맞는 제어 아키텍처를 신중하게 선택해야 합니다.
