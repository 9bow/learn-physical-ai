## 두 가지 환경 설계 패러다임

Isaac Lab은 로봇 학습 환경을 구현하는 두 가지 방식을 공식적으로 지원합니다. **Manager-based 환경**과 **Direct 환경**입니다. 두 방식 모두 동일한 Isaac Lab 인프라(GPU 병렬화, Isaac Sim 물리)를 활용하지만, 코드 구조와 설계 철학이 다릅니다.

선택은 프로젝트의 규모, 재사용성 요구, 개발자의 선호도에 따라 달라집니다.

## Manager-based 환경

### 구조

Manager-based 환경은 Isaac Lab의 **권장 패러다임**입니다. 환경을 구성하는 각 요소(관측, 행동, 보상, 종료)를 독립적인 매니저(Manager) 객체로 분리합니다.

```text
┌─────────────────────────────────────────────────────────────┐
│               Manager-based 환경 구조                        │
│                                                             │
│  ManagerBasedRLEnv                                          │
│  ├── ObservationManager                                     │
│  │     ├── term: joint_positions()                          │
│  │     ├── term: joint_velocities()                         │
│  │     └── term: base_lin_vel()                             │
│  │                                                          │
│  ├── ActionManager                                          │
│  │     └── term: JointPositionAction                        │
│  │                                                          │
│  ├── RewardManager                                          │
│  │     ├── term: track_lin_vel_xy (weight=+1.0)             │
│  │     ├── term: track_ang_vel_z (weight=+0.5)              │
│  │     └── term: action_rate (weight=-0.01)                 │
│  │                                                          │
│  ├── TerminationManager                                     │
│  │     ├── term: time_out()                                 │
│  │     └── term: base_height()                              │
│  │                                                          │
│  └── CurriculumManager (선택)                               │
│        └── term: terrain_levels()                           │
└─────────────────────────────────────────────────────────────┘
```

각 매니저는 여러 **term(항목)**으로 구성됩니다. term은 독립적인 함수 또는 클래스로, 매니저가 호출 시 각 term의 결과를 합산하거나 이어 붙입니다.

### 환경 설정(Config) 클래스

Manager-based 환경에서는 `ManagerBasedRLEnvCfg`를 상속한 설정 클래스가 환경 전체를 정의합니다.

```python
# 개념 이해용 의사코드 — 실제 API는 공식 문서 확인
from isaaclab.envs import ManagerBasedRLEnvCfg
from isaaclab.managers import ObservationGroupCfg, RewardTermCfg

class MyRobotEnvCfg(ManagerBasedRLEnvCfg):
    # 시뮬레이션 기본 설정
    decimation: int = 4          # 물리 스텝 / 정책 스텝 비율
    episode_length_s: float = 20.0  # 에피소드 최대 길이(초)

    # 관측 설정 (ObservationManager가 이 설정을 읽음)
    class ObservationsCfg:
        class PolicyObs(ObservationGroupCfg):
            joint_pos = ...
            joint_vel = ...

    # 보상 설정 (RewardManager가 이 설정을 읽음)
    class RewardsCfg:
        alive = RewardTermCfg(func=track_survival, weight=+1.0)
        action_rate = RewardTermCfg(func=penalize_action_rate, weight=-0.01)
```

설정 클래스는 Python 데이터클래스 스타일로 작성되며, 매니저는 이 설정을 읽어 자신의 term 목록을 구성합니다. 덕분에 YAML이나 JSON 없이 Python만으로 타입 안전한 설정 관리가 가능합니다.

### 장점

- **재사용성**: 개별 term(보상 함수, 관측 항목)을 여러 환경에서 공유할 수 있습니다.
- **가독성**: 환경의 구성 요소가 명확히 분리되어 있어 코드를 파악하기 쉽습니다.
- **실험 유연성**: 보상 항목의 가중치를 설정 클래스 한 곳에서 조정할 수 있습니다.
- **공식 지원**: Isaac Lab 공식 태스크 대부분이 Manager-based로 구현됩니다.

## Direct 환경

### 구조

Direct 환경은 단일 클래스 안에 관측, 보상, 종료 로직을 직접 구현합니다. Isaac Gym 스타일에 익숙한 개발자나 빠른 프로토타이핑이 필요할 때 적합합니다.

```text
┌─────────────────────────────────────────────────────────────┐
│               Direct 환경 구조                               │
│                                                             │
│  DirectRLEnv                                                │
│  ├── _setup_scene()       # 씬 구성                          │
│  ├── _pre_physics_step()  # 물리 스텝 전처리                 │
│  ├── _get_observations()  # 관측 수집 (직접 구현)            │
│  ├── _get_rewards()       # 보상 계산 (직접 구현)            │
│  ├── _get_dones()         # 종료 조건 (직접 구현)            │
│  └── _reset_idx()         # 환경 리셋 (직접 구현)            │
└─────────────────────────────────────────────────────────────┘
```

```python
# 개념 이해용 의사코드 — 실제 API는 공식 문서 확인
from isaaclab.envs import DirectRLEnv, DirectRLEnvCfg

class MyDirectEnv(DirectRLEnv):
    cfg: DirectRLEnvCfg

    def _setup_scene(self):
        # 로봇, 지형 등 씬 구성
        self.robot = Articulation(self.cfg.robot_cfg)
        self.scene.articulations["robot"] = self.robot

    def _get_observations(self) -> dict:
        # GPU 텐서로 직접 관측 구성
        obs = torch.cat([
            self.robot.data.joint_pos,
            self.robot.data.joint_vel,
        ], dim=-1)
        return {"policy": obs}

    def _get_rewards(self) -> torch.Tensor:
        # 보상 직접 계산
        return compute_my_reward(self.robot.data)

    def _get_dones(self) -> tuple[torch.Tensor, torch.Tensor]:
        terminated = self.robot.data.root_pos_w[:, 2] < 0.3
        time_out = self.episode_length_buf >= self.max_episode_length - 1
        return terminated, time_out
```

### 장점

- **단순성**: 소규모 프로젝트나 빠른 실험에서 코드량이 적습니다.
- **유연성**: 매니저 추상화에 맞추지 않고 원하는 로직을 자유롭게 구현합니다.
- **이식성**: Isaac Gym 경험이 있는 개발자가 빠르게 전환할 수 있습니다.

## 두 방식 비교

```text
┌─────────────────────────────────────────────────────────────────┐
│              Manager-based vs Direct 비교                        │
├─────────────────────────┬───────────────────┬───────────────────┤
│ 항목                    │ Manager-based     │ Direct            │
├─────────────────────────┼───────────────────┼───────────────────┤
│ 코드 구조               │ 분산 (매니저별)   │ 집중 (단일 클래스)│
│ 재사용성                │ 높음              │ 낮음              │
│ 초기 설정 복잡도        │ 높음              │ 낮음              │
│ 장기 유지보수           │ 용이              │ 어려움            │
│ 공식 태스크 지원        │ 대부분 이 방식    │ 일부 제공         │
│ Isaac Gym 유사도        │ 낮음              │ 높음              │
│ 실험 구성 유연성        │ 높음 (설정 클래스)│ 중간              │
│ 학습 곡선               │ 가파름            │ 완만              │
└─────────────────────────┴───────────────────┴───────────────────┘
```

## 어떤 방식을 선택해야 하는가

**Manager-based를 선택하는 경우:**
- 여러 태스크에서 공통 관측·보상 항목을 공유해야 할 때
- 팀 프로젝트에서 코드 가독성과 유지보수가 중요할 때
- Isaac Lab 공식 태스크를 기반으로 확장하려는 경우
- 커리큘럼 학습(CurriculumManager)을 적극 활용할 계획인 경우

**Direct를 선택하는 경우:**
- 빠른 프로토타이핑이 우선인 경우
- Isaac Gym 코드를 Isaac Lab으로 마이그레이션하는 경우
- 매니저 추상화가 불필요한 단순한 단일 태스크를 개발하는 경우
- 매우 특수한 관측·보상 구조로 매니저 인터페이스가 맞지 않는 경우

## 환경 등록

Isaac Lab은 Gymnasium 스타일의 환경 레지스트리를 사용합니다. 새로운 환경을 등록하면 `--task` 옵션으로 실행할 수 있습니다.

```python
# 개념 이해용 의사코드 — 실제 API는 공식 문서 확인
import gymnasium as gym

gym.register(
    id="My-Robot-Task-v0",
    entry_point="my_package.envs:MyRobotEnv",
    kwargs={"env_cfg_entry_point": MyRobotEnvCfg},
)
```

등록된 환경은 다음과 같이 실행합니다.

```bash
./isaaclab.sh -p scripts/reinforcement_learning/rsl_rl/train.py \
  --task My-Robot-Task-v0 \
  --num_envs 1024
```

## 소결

Manager-based 환경은 구조화된 대규모 프로젝트에, Direct 환경은 빠른 실험이나 Isaac Gym 마이그레이션에 적합합니다. 실제 Isaac Lab 공식 태스크(보행, 매니퓰레이션 등)는 대부분 Manager-based로 구현되어 있으므로, 장기적으로는 Manager-based 패러다임에 익숙해지는 것이 유리합니다. 다음 챕터에서는 관측, 보상, 종료 조건을 구체적으로 어떻게 정의하는지 살펴봅니다.
