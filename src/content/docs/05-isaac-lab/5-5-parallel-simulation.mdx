## 왜 병렬 시뮬레이션이 필요한가

강화학습은 데이터를 대량으로 소비합니다. 정책 경사 방법(Policy Gradient)의 대표 알고리즘인 PPO는 매 업데이트마다 수천~수만 개의 환경 전환(transition)을 필요로 합니다. 단일 환경에서 순차적으로 이 데이터를 수집하면 수일에서 수주의 시간이 소요됩니다.

CPU 기반 병렬 시뮬레이션(예: MuJoCo 멀티프로세스)은 수십 개 환경 수준에서 실용적입니다. 그러나 이 접근법은 프로세스 생성 오버헤드, 프로세스 간 통신(IPC) 비용, 그리고 학습 루프를 위한 GPU로의 데이터 전송이 병목이 됩니다.

Isaac Lab은 이 문제를 **모든 환경을 단일 GPU 위에서 동시에 실행**함으로써 해결합니다.

## GPU 병렬화의 핵심 원리

### 물리 시뮬레이션의 텐서화

PhysX 5(Isaac Sim의 물리 엔진)는 GPU 배치 처리를 직접 지원합니다. Isaac Lab에서 N개의 환경이 실행될 때, 물리 상태는 단일 GPU 텐서로 표현됩니다.

```text
┌─────────────────────────────────────────────────────────────────┐
│              N=4096 병렬 환경의 텐서 구조                        │
│                                                                 │
│  관절 위치 텐서:  shape = (4096, 12)   [모든 환경 × 관절 수]   │
│  ┌────┬────┬────┬────┬─────┬────┐                              │
│  │env0│env1│env2│env3│ ... │env4095│                            │
│  └────┴────┴────┴────┴─────┴────┘                              │
│                                                                 │
│  관절 속도 텐서:  shape = (4096, 12)                            │
│  ┌────┬────┬────┬────┬─────┬────┐                              │
│  │env0│env1│env2│env3│ ... │env4095│                            │
│  └────┴────┴────┴────┴─────┴────┘                              │
│                                                                 │
│  보상 텐서:       shape = (4096,)                               │
│  ┌────┬────┬────┬────┬─────┬────┐                              │
│  │ r0 │ r1 │ r2 │ r3 │ ... │ r4095│                            │
│  └────┴────┴────┴────┴─────┴────┘                              │
│                                                                 │
│  → 모든 연산이 GPU 위에서 배치(batch)로 동시 처리               │
└─────────────────────────────────────────────────────────────────┘
```

물리 엔진은 4096개 환경의 상태를 동시에 시뮬레이션하며, 각 환경은 서로 독립적으로 진행됩니다(충돌 없음). 이는 CUDA의 병렬 스레드 모델과 잘 맞습니다.

### 학습 루프 전체의 GPU 상주

Isaac Lab의 진정한 강점은 물리 시뮬레이션만이 아니라 **학습 루프 전체**가 GPU에 상주한다는 점입니다.

```text
┌──────────────────────────────────────────────────────────────┐
│              학습 루프 내 데이터 이동                          │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                    GPU 메모리                         │    │
│  │                                                       │    │
│  │  PhysX 5          Isaac Lab          RL 알고리즘      │    │
│  │  ┌─────────┐      ┌──────────┐      ┌──────────┐    │    │
│  │  │  물리   │─────▶│관측 수집 │─────▶│PPO 업데이트│   │    │
│  │  │시뮬레이션│      │보상 계산 │      │(PyTorch) │    │    │
│  │  └────┬────┘      └──────────┘      └────┬─────┘    │    │
│  │       ▲                                   │          │    │
│  │       └───────────────────────────────────┘          │    │
│  │              행동 텐서 (action tensor)                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  CPU ↔ GPU 복사: 최소화 (로깅, 체크포인트 시에만)            │
└──────────────────────────────────────────────────────────────┘
```

CPU-GPU 데이터 전송(memcpy)은 매우 비용이 높습니다. Isaac Lab은 관측, 행동, 보상 텐서를 GPU에서 직접 생성하고 소비하므로 이 비용을 거의 제거합니다.

## 병렬 환경 실행 예시

`num_envs` 파라미터 하나로 병렬 환경 수를 조정할 수 있습니다.

```bash
# 64개 병렬 환경 (개발·디버깅용)
./isaaclab.sh -p scripts/reinforcement_learning/rsl_rl/train.py \
  --task Isaac-Velocity-Rough-Anymal-C-v0 \
  --num_envs 64

# 4096개 병렬 환경 (대규모 학습용, 고성능 GPU 필요)
./isaaclab.sh -p scripts/reinforcement_learning/rsl_rl/train.py \
  --task Isaac-Velocity-Rough-Anymal-C-v0 \
  --num_envs 4096 \
  --headless
```

환경 인스턴스들은 물리적으로 씬 안에 일정 간격으로 배치됩니다. Isaac Sim은 각 환경을 서로 다른 USD 프림(prim)으로 관리하며, 물리 경계가 겹치지 않도록 자동 오프셋을 적용합니다.

```text
┌─────────────────────────────────────────────────────────────┐
│           Isaac Sim 씬 내 병렬 환경 배치 (위에서 본 뷰)      │
│                                                             │
│  [env_0]  [env_1]  [env_2]  [env_3]  [env_4]  ...          │
│  [env_8]  [env_9]  [env_10] [env_11] [env_12] ...          │
│  [env_16] [env_17] ...                                      │
│                                                             │
│  ← env_spacing 파라미터로 간격 설정 →                       │
│  각 환경은 독립적인 물리 영역에서 실행                       │
└─────────────────────────────────────────────────────────────┘
```

## 샘플 처리량의 의미

**샘플 처리량(sample throughput)**은 초당 수집되는 환경 전환(환경 스텝) 수입니다. 이 지표는 RL 학습 속도의 핵심 결정 요인입니다.

```text
샘플 처리량 = num_envs × (1 / dt) × 정책 실행 속도 계수
```

예를 들어, 4096개 환경, dt=0.005s (200Hz 물리), decimation=4 (50Hz 정책)에서 실시간 처리 기준:

```
이론적 처리량 = 4096 × 50 = 204,800 스텝/초
```

환경 수만큼 샘플을 동시에 수집할 수 있습니다. 실제로는 GPU 병렬 처리 효율과 메모리 대역폭에 따라 이론값의 60~90% 수준이 일반적입니다.

아래 표는 하드웨어·태스크에 따라 크게 달라지는 예시값이며, 절대값보다 상대적 차이의 규모를 참고하십시오.

| 설정 | 처리량(예시, 하드웨어·태스크에 따라 다름) | 비고 |
|---|---|---|
| CPU, 단일 환경 | ~100 스텝/초 | MuJoCo 단일 |
| CPU, 32 환경 | ~3,000 스텝/초 | 멀티프로세스 |
| Isaac Lab, 64 환경 | ~50,000+ 스텝/초 | RTX 3090 기준 예시 |
| Isaac Lab, 4096 환경 | ~500,000+ 스텝/초 | A100 80GB 기준 예시 |

## 리셋 처리: 에피소드별 독립성

병렬 환경에서 각 환경은 독립적으로 종료되고 리셋됩니다. 환경 0이 넘어져서 종료되어도 환경 1은 계속 실행됩니다.

```python
# 개념 이해용 의사코드 — 실제 API는 공식 문서 확인
def step(self, actions):
    # 모든 환경에 행동 적용
    self.robot.set_joint_effort_target(actions)

    # 물리 스텝 (N번 반복, decimation)
    for _ in range(self.decimation):
        self.sim.step()

    # 관측/보상/종료 계산 (배치 처리)
    obs = self._get_observations()       # shape: (N, obs_dim)
    rewards = self._get_rewards()        # shape: (N,)
    terminated, timed_out = self._get_dones()  # shape: (N,)

    # 종료된 환경만 선택적으로 리셋
    reset_env_ids = (terminated | timed_out).nonzero().squeeze(-1)
    if len(reset_env_ids) > 0:
        self._reset_idx(reset_env_ids)

    return obs, rewards, terminated, timed_out, {}
```

선택적 리셋(selective reset)은 중요한 효율성 요소입니다. 전체 배치를 멈추고 동기화하지 않아도 되므로 GPU 활용률을 높게 유지할 수 있습니다.

## num_envs 선택 가이드

GPU VRAM과 태스크 복잡도에 따라 적절한 `num_envs`를 선택해야 합니다.

아래는 하드웨어·태스크에 따라 크게 달라지는 예시값입니다.

| GPU VRAM | 권장 num_envs 예시 (일반 보행 태스크 기준) |
|---|---|
| 8 GB | 256 ~ 512 |
| 16 GB | 1024 ~ 2048 |
| 24 GB | 2048 ~ 4096 |
| 40+ GB | 4096 ~ 8192+ |

VRAM이 부족하면 Isaac Sim이 OOM(Out of Memory) 오류를 발생시킵니다. 처음에는 작은 값(64 또는 128)으로 시작해서 점진적으로 늘리는 방법을 권장합니다.

## 소결

Isaac Lab의 대규모 병렬 시뮬레이션은 GPU 텐서 기반의 물리 시뮬레이션, 학습 루프 전체의 GPU 상주, 그리고 선택적 리셋 메커니즘의 조합으로 구현됩니다. 수천 개의 환경을 동시에 실행하여 단일 환경 대비 수천 배의 샘플 처리량을 달성하며, 이는 복잡한 로봇 태스크를 현실적인 시간 내에 학습할 수 있게 하는 핵심 기반입니다. 다음 챕터에서는 이 인프라 위에서 실제 RL 알고리즘을 연결하는 방법을 살펴봅니다.
